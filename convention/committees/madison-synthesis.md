## 2. Areas of Strong Consensus (15+ Delegates)

These are constitutional slam-dunks — virtually no opposition, broad cross-role support.

### 2.1 API Reference Documentation (~22 delegates)
The single most-mentioned document type. Every role from frontend to backend to QA to platform recognizes API references as essential. Near-universal agreement that machine-readable formats (OpenAPI) should be the source of truth, with human-readable views generated from them.

**Champions include:** Spaight, Brearly, Hamilton, Williamson, CPinckney, McHenry, GMorris, FitzSimons, Franklin, Madison, Sherman, Washington, WSJohnson, and more.

### 2.2 Runbooks & Operational Procedures (~20 delegates)
The second most-cited document type. Runbooks are endorsed by operations, infrastructure, SRE, DBA, security, cloud, support, architecture, and leadership roles. Universal agreement that runbooks must be simultaneously human-readable (for sleep-deprived engineers at 3 AM) and machine-parseable (for agent execution).

**Champions include:** RMorris, Carroll, Few, Livingston, CCPinckney, Dickinson, King, Sherman, Washington, WSJohnson, Franklin, Read, Wythe, and more.

### 2.3 Documentation Style Guides & Standards (~20 delegates)
Nearly every role sees the need for codified documentation standards — whether as coding standards, documentation style guides, naming conventions, or conventions files. The debate is not whether but what form they take.

**Champions include:** Madison, Jenifer, Mason, Lansing, Franklin, King, Sherman, Williamson, Dayton, CCPinckney, Blount, Broom, Bedford, Bassett, and more.

### 2.4 Architecture Decision Records (~18 delegates)
The strongest consensus around any single *authored* (non-generated) document type. Delegates from every layer — backend, frontend, ops, architecture, compliance, domain — champion ADRs. Endorsed by both 3-year and 35-year veterans.

**Champions include:** Spaight, CCPinckney, Dickinson, FitzSimons, GMorris, Hamilton, Houston, Ingersoll, King, Madison, McHenry, Sherman, Washington, WSJohnson, Wythe, Franklin, Paterson, Dayton.

### 2.5 README / Getting Started / Onboarding (~17 delegates)
The "front door" document. Every delegate who mentions onboarding identifies the README or getting-started guide as the highest-leverage single document. Cross-role agreement that these must be deterministic, step-by-step, and tested.

**Champions include:** AMartin, CPinckney, Franklin, Williamson, King, Washington, Sherman, McHenry, WSJohnson, Dayton, Mifflin, Bassett, Bedford, Broom, Brearly, Mason.

### 2.6 Structured Metadata / Frontmatter on Every Document (~25+ delegates)
The most universal *structural* requirement. Delegates converge on requiring YAML frontmatter or equivalent metadata on every document: type, owner, audience, last-reviewed date, status, and related documents. Endorsed by balanced, human, and agent perspectives alike.

**Champions include:** CCPinckney, Madison, McClurg, Wythe, Washington, Rutledge, Ellsworth, Jenifer, Mason, Williamson, Broom, Strong, Bassett, Few, Hamilton, Langdon, and many more.

### 2.7 Freshness Tracking & Staleness Prevention (~22+ delegates)
Every delegate who discusses anti-patterns mentions stale documentation as a primary failure mode. Strong consensus that every document must carry a freshness indicator and that decay should be actively monitored.

**Key voices:** Few ("docs should have SLOs"), Madison ("lifecycle policy"), Wythe ("version history"), RMorris ("freshness requirements"), Livingston ("bus factor"), Gorham ("CI validation"), and many more.

### 2.8 Documentation Must Serve Both Humans and Agents (~50+ delegates)
With the possible exception of 2-3 delegates, every report explicitly addresses dual-audience documentation. This is the meta-consensus of the Convention: the Docstitution must govern documentation for both human and machine consumers.

### 2.9 Documentation-as-Code / Colocation with Source (~18+ delegates)
Strong agreement that documentation should live in version control alongside the code it describes, be reviewed in pull requests, and be treated as a development artifact.

**Key voices:** Sherman ("put docs where developers look"), WSJohnson ("colocate or accept obsolescence"), Hamilton ("derive, don't duplicate"), FitzSimons ("reviewed in PRs"), McHenry ("in-source over wikis"), Mifflin ("doc-tests"), GMorris ("IDE-discoverable").

### 2.10 Glossary / Shared Vocabulary (~15+ delegates)
Cross-role agreement on the need for controlled vocabulary. Endorsed by business analysts, domain experts, data analysts, technical writers, support leads, architects, and user documentation specialists.

**Champions include:** Franklin, AMartin, CPinckney, Pierce, Wilson, Houston, Clymer, Jenifer, Butler, Madison, Gilman, Lansing, Brearly, Mason, Dayton.

## 4. Areas of Contention

### 4.1 The Great Authorship Debate: Auto-Generated vs. Human-Authored

**Auto-generation maximalists:**
- Gorham: "If a machine can generate a document, a human should not write it."
- Hamilton: "Derived documentation is superior to authored documentation wherever derivation is possible."
- Mercer: "The best data documentation is not written — it is emitted."
- Butler: "The only database documentation you can trust is generated from the database itself."

**Human authorship advocates:**
- Franklin: "I worry that in our rush to make documents machine-readable, we will strip away the narrative, the context, the *humanity*."
- Gerry: "Agents generate docs at unprecedented speed — meaning inaccurate docs proliferate at unprecedented speed."
- LMartin: "Gherkin must remain human-authored to capture meaningful business behavior."
- Yates: "Bug reports need human narrative; the tester's account is irreplaceable."

**Compromise position (majority):** Most balanced delegates agree on a division: auto-generate structural/factual docs (API refs, schema docs, changelogs, environment matrices) while preserving human authorship for intent, rationale, tutorials, and narrative.

### 4.2 Minimalists vs. Comprehensive Documenters

**Minimalists:**
- McHenry: "Every required document type must justify its existence by answering a question someone actually asks."
- Sherman: "Resist the temptation to mandate documentation volume and instead focus on documentation value."
- Paterson: "If a document doesn't help ship faster or safer, eliminate it."
- Ellsworth: "Does the team actually use it within the sprint cadence?"

**Comprehensive advocates:**
- Madison: "A comprehensive taxonomy that classifies every document type by purpose, audience, and lifecycle stage."
- CCPinckney: "Every document must carry structured frontmatter… frontmatter is not optional metadata — it is the navigation system."
- Wythe: "Traceability, version control, and change authorization as non-negotiable requirements for every document."
- Washington: "A hierarchical framework where every document has a defined type, a clear owner, a bounded scope."

**Fault line:** Minimalists fear bureaucratic overhead stifling velocity; comprehensive advocates fear under-governance creating chaos. The tension is about *how much structure is too much*.

### 4.3 Code-as-Documentation vs. Prose Documentation

**Code-as-docs advocates:**
- GMorris: "The type system is the most reliable documentation."
- Mifflin: "Documentation should be held to the same standard as code: if it passes tests, it ships."
- Gorham: "The pipeline IS the documentation."
- Carroll: "Terraform files ARE documentation."
- Hamilton: "When IntelliSense renders XML doc comments alongside code, we achieve documentation structurally incapable of drifting."

**Prose champions:**
- Franklin: "Structure and machine-readability are the skeleton, not the soul."
- AMartin: "Documentation must meet users where they are… tutorials teach."
- Yates: "The ultimate test of any document is whether a human can use it without help."
- Mason: "Documentation has users. Docs are products with their own user experience."

**Resolution path:** Nearly all delegates accept that both are needed. The Docstitution should define *tiers* of documentation reliability, with executable/tested docs as the highest tier and narrative as complementary.

### 4.4 Where Docs Should Live: Colocated vs. Centralized

**Colocators:**
- Sherman: "Put documentation where developers look — in the code, in the repository."
- WSJohnson: "Colocate or accept obsolescence."
- McHenry: "Prefer in-source documentation over external wikis."
- GMorris: "Documentation outside the development environment gets ignored."

**Centralists:**
- Madison: "Taxonomy & Information Architecture Specification — the authoritative map."
- Mason: "Information Architecture Map — every author and agent uses this to place content."
- Gilman: "Knowledge Base Articles — canonical support source."
- Jenifer: "Documentation outside the pipeline will rot unchecked."

**Resolution path:** Not a true either/or. Colocation for code-adjacent docs (ADRs, READMEs, inline comments); centralized architecture for cross-cutting docs (style guides, glossaries, governance). The taxonomy bridges both.

### 4.5 Agent Access Scope: Open vs. Tiered

**Open access:**
- McClurg: "Every document should be structured enough for agents to parse programmatically."
- Dayton: "Optimizing for agents is an improvement for everyone."
- CCPinckney: "Make documentation machine-traversable by default."

**Tiered / restricted:**
- Read: "Security docs need access tiers — threat models and vulnerability details require classification levels preventing agents from exposing sensitive details."
- Wythe: "Agents must be identified as distinct actors in audit trails."
- Gerry: "Human reviewers must stay in the loop. The Docstitution must require human sign-off."

**Resolution path:** Access classification system. Most docs open to agents; security-sensitive docs (threat models, vulnerability details, secrets) require access tiers. All agent actions auditable.

### 4.6 Visual Documentation vs. Plain-Text Primacy

**Visual-first:**
- WSJohnson: "I will draw a diagram before I write a paragraph. Architecture is spatial."
- Blount: "Frontend documentation is fundamentally visual. Requiring only markdown is like requiring only sheet music."
- Mason: "Storybook snapshots, usability test reports — visual evidence matters."

**Plain-text-first:**
- Bedford: "The best documentation can be read with `cat` and `less`. No doc should require a browser."
- McHenry: "Documentation should be as lean as the language it describes."
- Sherman: "Keep documents short enough to actually read."

**Resolution path:** Diagram-as-code (Mermaid, PlantUML, Structurizr DSL) satisfies both camps — text-based source in version control generating visual output.

## 5. Proposed Article Structure

### Preamble: Purpose, Scope, Philosophy

**Scope:** Establishes that the Docstitution governs documentation for software products and systems, addressing both human and AI agent consumers.

**Core principles to encode:**
1. Documentation is infrastructure, not overhead
2. Every document must declare what it is, who it is for, and when it was last verified
3. Documentation serves both human and machine audiences from shared sources
4. Structure enables, rather than burdens, documentation creators

**Drafting committee:** Washington (president), Franklin (philosophical voice), Sherman (pragmatist), Madison (architecture)

---

### Article I: Foundational Principles

**Scope:** The non-negotiable values underpinning all other Articles.

**Principles derived from consensus:**
1. **Dual-audience primacy** — Every document must be consumable by both humans and agents (50+ delegates)
2. **Proximity** — Documentation lives with the artifact it describes (18+ delegates)
3. **Freshness as quality** — A stale document is worse than a missing one (22+ delegates)
4. **Verifiability** — Documentation should be testable and verifiable where possible (Gerry, LMartin, Mifflin, Few)
5. **Readability** — A document that cannot be understood by its intended audience has failed (Franklin, Sherman, Mason, AMartin)
6. **Ownership** — Every document must have a named, current steward (Washington, Livingston, Madison, Wythe)
7. **Accessibility** — Documentation must be accessible to people with disabilities and to constrained environments (Lansing, Yates, Blount, Bedford, Broom)

**Document types mapped:** Documentation Charter, Governance Charter

**Drafting committee:** Franklin (readability), Lansing (accessibility), Few (freshness/SLOs), Washington (ownership), Sherman (proximity), CCPinckney (dual-audience structure)

---

### Article II: Document Types & Taxonomy

**Scope:** The authoritative classification of all document types, organized by purpose, audience, and lifecycle characteristics.

**Taxonomy tiers (proposed):**
1. **Executable / Verified** — Tests, doc-tests, OpenAPI specs validated in CI (highest reliability)
2. **Generated / Derived** — Auto-generated from code, config, schema, or telemetry
3. **Authored / Maintained** — Human-written narrative, rationale, tutorials, guides
4. **Ephemeral / Ceremonial** — Sprint artifacts, meeting minutes, status reports (shortest lifecycle)

**Document types mapped:** Full catalog from Section 1 above, organized into the taxonomy

**Drafting committee:** Madison (taxonomy architecture), Jenifer (templates), CCPinckney (classification standards), Brearly (API-specific types), RMorris (operational types), Pierce (domain types), Randolph (product types), Wythe (compliance types)

---

### Article III: Lifecycle Governance

**Scope:** When and how documents are created, reviewed, updated, deprecated, and archived.

**Key provisions:**
- Creation triggers (new service → runbook, new API → spec, new decision → ADR)
- Review cadences per document tier
- Staleness thresholds and decay warnings
- Deprecation and archival procedures
- Version history and change rationale requirements
- Definition of Done must include documentation currency

**Document types mapped:** Document Lifecycle Policy, Content Audit Checklists, Freshness SLOs, Retention & Disposal Policies

**Drafting committee:** Madison (lifecycle design), Ellsworth (DoD integration), Few (freshness SLOs), Wythe (retention/audit), Paterson (cadence management), Livingston (ownership transfer)

---

### Article IV: Structure & Format Standards

**Scope:** How documents are organized, formatted, and made navigable for both humans and agents.

**Key provisions:**
- Mandatory structured metadata headers (YAML frontmatter)
- Semantic heading hierarchy (no skipping levels)
- Explicit, typed cross-references between documents
- Dual-format principle: structured data for agents alongside narrative for humans
- Accessibility requirements: WCAG 2.2 AA, alt text, semantic HTML
- Platform-aware content: conditional rendering, platform tags
- Plain-text-friendly as baseline; rich rendering as enhancement
- Diagram-as-code for all visual documentation

**Document types mapped:** Style Guides, Templates, Linting Rulesets, Information Architecture Specs, Accessibility Standards

**Drafting committee:** Jenifer (style/consistency), CCPinckney (structural standards), Lansing (accessibility), Bedford (plain-text), Bassett (platform parity), WSJohnson (diagrams-as-code), Mason (information architecture), Broom (mobile/responsive)

---

### Article V: Roles & Responsibilities

**Scope:** Who creates, reviews, owns, and maintains documentation — including the role of AI agents.

**Key provisions:**
- Every document must have a named human owner
- Agents as documentation participants: generators, validators, enforcers — but not sole authors of narrative docs
- Human review gates for quality, accuracy, and meaning
- Ownership transfer procedures when personnel change
- Documentation responsibilities in Definition of Done
- Role taxonomy: Author, Reviewer, Owner, Consumer (Human), Consumer (Agent)
- Agent identity in audit trails: agents must be identified as distinct actors
- Access tiers for security-sensitive documentation

**Document types mapped:** RACI Matrix, Governance Charter, Access Control Policies, Review & Approval Workflows

**Drafting committee:** Gerry (human review gates), Washington (governance), Wythe (audit/accountability), King (team responsibilities), Read (access tiers), McClurg (agent roles), Gilman (consumer advocacy)

---

### Article VI: Tooling & Infrastructure

**Scope:** The technical infrastructure supporting documentation creation, validation, and delivery.

**Key provisions:**
- Documentation-as-code: docs in version control, reviewed in PRs
- CI/CD for documentation: linting, link checking, freshness validation, build gates
- Auto-generation pipelines: schema docs, API refs, changelogs, environment matrices
- Terminology registry as queryable service
- Doc-tests and executable examples validated in CI
- Multiple delivery formats from single source (web, offline bundles, terminal-friendly)
- Search and discoverability infrastructure

**Document types mapped:** Doc Pipeline Configurations, Linting Rulesets, Document Schema Registry, Build System Docs

**Drafting committee:** Gorham (CI/CD pipelines), Hamilton (auto-generation), Jenifer (linting/pipelines), Mifflin (doc-tests), Mercer (self-documenting systems), Carroll (infrastructure docs), Williamson (platform delivery)

---

### Article VII: Amendment Process

**Scope:** How the Docstitution itself evolves over time.

**Key provisions:**
- Proposal process for new document types or standard changes
- Evidence requirements for amendments (demonstrated need, not theoretical)
- Review by affected roles before ratification
- Versioning of the Docstitution itself
- Sunset clauses: provisions that must be reaffirmed periodically
- Retrospective feedback loops: practitioner experience informs amendments

**Drafting committee:** Washington (process authority), Madison (structural integrity), Ellsworth (iterative improvement), Paterson (change management), Sherman (pragmatic review)

## 7. Key Tensions to Resolve

### Tension 1: Human Readability vs. Machine Parseability
**Question:** When structure and narrative conflict, which takes priority?

| Side | Delegates | Position |
|---|---|---|
| **Human-first** | Franklin, Mason, AMartin, Yates, Gilman, Gerry, Lansing | Narrative clarity is primary; structure serves as skeleton, not soul |
| **Agent-first** | CCPinckney, McClurg, Gorham, Hamilton, Dayton, Houston, Mercer, Mifflin | Structured, machine-traversable documentation enables both audiences |
| **Bridge position** | Sherman, Washington, Madison, Williamson | Lightweight structure that serves agents without burdening human authors |

**Resolution recommendation:** Adopt the "dual-layer" principle proposed by multiple delegates. Require structured metadata headers and semantic organization (low burden, high agent value). Preserve narrative sections within that structure. Structure is the frame; narrative fills it.

---

### Tension 2: Documentation Volume — Minimal Viable vs. Comprehensive
**Question:** How many document types should the Docstitution mandate?

| Side | Delegates | Position |
|---|---|---|
| **Minimalists** | McHenry, Sherman, Paterson, Ellsworth | Fewer docs, rigorously maintained; every doc must justify its existence |
| **Comprehensivists** | Madison, CCPinckney, Wythe, Washington | Complete taxonomy; structural discipline prevents chaos |

**Resolution recommendation:** Mandate a *small core* of required document types per project (README, ADR, API spec, runbook, style guide, glossary). Provide a comprehensive catalog of *recommended* types that teams adopt based on context. Require justification for omitting core types rather than for adding optional ones.

---

### Tension 3: Auto-Generated vs. Human-Authored Content
**Question:** What should agents write and what should humans write?

| Side | Delegates | Position |
|---|---|---|
| **Maximize generation** | Gorham, Hamilton, Mercer, Butler, Mifflin | Generate everything derivable from code/config/telemetry; humans write only "why" |
| **Protect human authorship** | Franklin, Gerry, Yates, AMartin, LMartin | Humans must author tutorials, rationale, narrative; agents assist but don't replace |

**Resolution recommendation:** Establish a clear division: agents generate factual/structural content (API refs, schema docs, changelogs, env configs); humans author intent/rationale content (ADRs, tutorials, narratives, design docs). Require human review of all agent-generated content before publication.

---

### Tension 4: Colocation vs. Central Governance
**Question:** Should docs live in each repository or in a governed central system?

| Side | Delegates | Position |
|---|---|---|
| **Colocators** | Sherman, WSJohnson, McHenry, GMorris, Mifflin, Hamilton | Docs in the repo, versioned with code, or they're already dead |
| **Centralists** | Madison, Mason, Jenifer, Gilman, Langdon | Cross-cutting docs need central governance, taxonomy, and discovery |

**Resolution recommendation:** Both. Code-adjacent docs (READMEs, ADRs, inline comments, API specs) colocate in repositories. Cross-cutting docs (style guides, glossaries, governance, architecture overviews) live in a governed central location. A taxonomy and cross-reference index bridges the two.

---

### Tension 5: Agent Access — Open Access vs. Security Tiers
**Question:** Should agents have unrestricted access to all documentation?

| Side | Delegates | Position |
|---|---|---|
| **Open access** | McClurg, CCPinckney, Dayton, Gorham | Agents need full traversal; restriction limits their value |
| **Tiered access** | Read, Wythe, Gerry | Threat models, vulnerability data, and secrets require access classification |

**Resolution recommendation:** Default to open access. Establish a classification system for sensitive docs (threat models, vulnerability details, credential procedures). Require agents to respect access tiers. Log all agent access to sensitive docs in audit trails.

---

### Tension 6: Executable Specs vs. Prose Specifications
**Question:** Should passing tests replace written specifications?

| Side | Delegates | Position |
|---|---|---|
| **Tests-as-truth** | LMartin, Mifflin, GMorris | Passing tests are verified docs; prose can lie; tests cannot |
| **Prose-as-context** | Franklin, AMartin, Houston, Pierce | Tests prove behavior; prose explains *why*, provides mental models, teaches |

**Resolution recommendation:** Establish a documentation reliability hierarchy: (1) executable/tested specs are the highest-fidelity documentation, (2) auto-generated docs from source are next, (3) authored prose provides context and rationale. When prose and tests disagree, tests tell truth about current behavior; prose tells truth about intent.

---

### Tension 7: Process Weight — Sprint-Friendly vs. Compliance-Ready
**Question:** How much documentation process is too much?

| Side | Delegates | Position |
|---|---|---|
| **Lightweight** | Ellsworth, McHenry, Paterson | Docs must fit sprint cadence; heavy specs break velocity |
| **Rigorous** | Wythe, Read, Washington | Traceability, audit trails, and change control are non-negotiable |

**Resolution recommendation:** Tier the process by document class. Sprint artifacts (backlogs, standup notes) get lightweight treatment. Regulated artifacts (compliance matrices, audit trails) get full change control. Most docs fall in between: reviewed in PRs, owned, with freshness tracking.

---

### Tension 8: Visual Documentation — Rich vs. Plain
**Question:** Should the Docstitution privilege any rendering format?

| Side | Delegates | Position |
|---|---|---|
| **Visual-first** | WSJohnson, Blount, Mason | Architecture is spatial; component docs need visual previews |
| **Plain-text-first** | Bedford, McHenry, Sherman | Best docs work in a terminal; no browser should be required |

**Resolution recommendation:** Mandate diagram-as-code (Mermaid, PlantUML, Structurizr DSL) for architectural visuals — text-based source files that generate rendered output. Require alt text and text descriptions for all visual content. Blount's Storybook-specific needs can be accommodated as a "living doc" format alongside the plain-text baseline.
